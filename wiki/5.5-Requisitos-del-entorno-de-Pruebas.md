# Configuración del entorno de pruebas

Este documento detalla la configuración del entorno necesario para la ejecución de las pruebas unitarias.

## Infraestructura de CI/CD

Se utilizará GitHub Actions como plataforma de integración continua.

- **Runners:** Se utilizarán runners de GitHub con el sistema operativo `ubuntu-24.04`.
- **Estrategia de Ramas:** Se sigue una estructura de ramas optimizada para desarrollo ágil: `main` (producción/estable) y ramas de tarea categorizadas como `category/task` (ej. `feat/login`, `fix/bug-123`).
- **Disparadores (Triggers):**
  - **Pull Requests:** La suite completa de pruebas unitarias se ejecuta automáticamente ante la creación o actualización de cualquier Pull Request hacia la rama `main`.
  - **Push a Main:** Tras la aprobación y mezcla (merge) en `main`, se dispara un flujo de construcción que genera y publica la imagen del contenedor en el GitHub Container Registry (GHCR).

## Workflow

El flujo de trabajo para las pruebas unitarias y entrega continua sigue las siguientes etapas:

1. **Validation (PR):**
   - Compilación del código fuente.
   - Ejecución de la suite de pruebas unitarias (JUnit 5).
   - Verificación de cobertura con JaCoCo (Target $\ge 85\%$).
   - Requiere aprobación manual del Test Leader (Christian Mestas).
2. **Delivery (Main):**
   - Construcción de imagen Docker.
   - Publicación en GHCR.

## Configuración de Software y Entorno

Para replicar el entorno de pruebas localmente o en el CI, se requieren los siguientes componentes:

### Requisitos de Software
- **Java JDK 17:** Distribución Temurin.
- **Gradle:** Utilizado como herramienta de construcción (gestión de dependencias y ejecución de tareas).
- **Node.js:** Versión 22.x (gestionada automáticamente por el plugin de Gradle para las tareas de frontend).
- **Docker:** Para la ejecución de **Testcontainers**.

### Base de Datos y Servicios
- **PostgreSQL:** Las pruebas utilizan contenedores efímeros de PostgreSQL gestionados por Testcontainers. Se soportarán oficialmente las versiones 10, 15 y 16.
- **Testcontainers:** Facilita la creación y destrucción automática de los servicios necesarios, garantizando la **replicabilidad** de las pruebas en cualquier entorno que soporte Docker.

### Procedimientos de Prueba
- **Setup:** El entorno se configurará automáticamente mediante el comando `./gradlew build`.
- **Ejecución:** Las pruebas se lanzarán como parte del ciclo de vida de Gradle.
- **Limpieza:** Los contenedores de Docker se eliminarán automáticamente al finalizar las pruebas.
- **Reportes:** Los resultados de las pruebas se generarán en formato XML y HTML.
